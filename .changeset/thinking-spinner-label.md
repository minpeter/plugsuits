Surface reasoning/thinking on the foreground spinner, extract a shared spinner primitive, and render the tool-pending `Executing...` indicator by default so consumers without custom tool renderers (e.g. `minimal-agent`) see it automatically.

- TUI: while the model streams reasoning parts, the foreground spinner now stays visible with the label swapped to `Thinking...`. Reasoning parts are treated as non-visible for the first-visible-part trigger, so the spinner is only cleared when real text/tool output arrives. `PiTuiStreamState` gains optional `onReasoningStart` / `onReasoningEnd` callbacks; a new `handleReasoningEnd` handler fires `onReasoningEnd` (previously `reasoning-end` was silently dropped via `IGNORE_PART_TYPES`). `renderAgentStream` wires these into the spinner so the label temporarily becomes `Thinking...` during reasoning spans and restores the caller-provided base label (`Working...`) when reasoning ends.
- TUI: fix `Thinking...` label missing on the second reasoning span of a turn (post tool-call). When the continuation stream emits a visible part (`text-start`, `tool-result`, etc.) before the `reasoning-start`, `clearStreamingLoader` nulls the foreground spinner first, so the subsequent `foregroundStatus?.setMessage("Thinking...")` silently no-ops. `onReasoningStart` now revives the spinner via `showLoader("Thinking...")` if it was cleared, and `onReasoningEnd` tears down only the spinner it revived (so ordinary flows that kept the base `Working...` loader alive still restore that label unchanged).
- TUI: new shared primitive `@ai-sdk-tool/tui` → `pending-spinner.ts` exposing `PENDING_SPINNER_FRAMES`, `PENDING_SPINNER_INTERVAL_MS`, `stylePendingIndicator(frame, message)`, and `createSpinnerTicker(onFrame, options?)`. `StatusSpinner`, `FooterStatusBar`, `BaseToolCallView`'s tool-pending block, and the internal `pi-tui-stream-renderer.ts` tool view in CEA all route through this primitive, so any future palette/cadence/glyph change lands in one place.
- TUI: show `Executing...` on the foreground status spinner (the same slot as `Processing...` / `Working...` / `Thinking...`) while a tool call is in flight. The earlier attempt rendered an inline `Executing...` `Text` child inside `BaseToolCallView` (a chatContainer descendant), which sat far from the editor with the idle placeholder's two blank lines (`IdleStatusPlaceholder.render() === ["", ""]`) separating it from the prompt. The inline indicator was removed and replaced with `onToolPendingStart` / `onToolPendingEnd` callbacks on `PiTuiStreamState`, fired from `handleToolCall` / `handleToolResult` / `handleToolError` / `handleToolOutputDenied`. `renderAgentStream` wires these into the same foreground spinner slot (`showLoader("Executing...")` with revive-if-null semantics, then restores the base `Working...` label on result) using a counter so parallel tool calls only clear the spinner when every pending call has resolved. `Executing...` now sits directly above the prompt, identical in placement to `Thinking...`.
- TUI: `handleToolResult` now fires `onToolPendingEnd` unconditionally (regardless of the `showToolResults` flag) so the spinner is always restored — the flag still controls whether the tool result renders visually, but it must not strand the spinner.
- TUI: remove the pretty-block internal pending spinner entirely. `setPrettyBlock({ isPending: true })` used to paint a rotating `Executing...` frame into `readBody` via `startPendingSpinner`, which duplicated the new foreground `Executing...` spinner. `setPrettyBlock` now simply sets the body to `""` when pending; the foreground spinner owns the pending affordance. All the pretty-block spinner plumbing (`startPendingSpinner`, `stopPendingSpinner`, `paintPendingBodyFrame`, `pendingSpinnerTicker`, `pendingTemplate`, `lastPendingFrame`, `PENDING_MESSAGE`, `PENDING_MARKER`) was deleted.
- TUI: fix the 2-blank-line gap above the foreground spinner during pretty-block pending state. `ensurePrettyBlockComponents` previously added a standalone `new Spacer(1)` between `readHeader` and `readBody`, which always emitted `[""]` even when `readBody` rendered to `[]` (empty-text short-circuit). Combined with `StatusSpinner.render()`'s own leading `""`, callers saw two blank rows above `Executing...`. The explicit `Spacer(1)` is removed from the pretty block; `BackgroundBody.render()` now prepends its own leading `""` only when the body has content, which preserves the original 1-blank separator between header and body in non-pending mode and eliminates the stray blank in pending mode. `Executing...` now sits with a single leading blank line, identical to `Thinking...` / `Working...`. Two regression tests added in `tool-call-view.test.ts`: pending pretty block has no trailing blank, non-pending pretty block keeps the header-body separator.
- TUI: `BaseToolCallView`'s existing pretty-block `Executing..` indicator adopts the canonical braille frames (`⠋⠙⠹⠸⠼⠴⠦⠧⠇⠏`) with cyan-frame / dim-message styling, and the label is normalized to `Executing...` (three dots, matching `Processing...` / `Working...` / `Thinking...`). Previously tool-pending blocks used plain ASCII `- \ | /` frames with no color.
- CEA: the internal `pi-tui-stream-renderer.ts` pending spinner is kept in sync via the shared primitive. Tests updated accordingly (`EXECUTING_SPINNER_TEXT_REGEX` matches braille frames; string assertions updated to `Executing...`). New `tool-call-view.test.ts` covers the automatic fallback indicator (appears while pending, cleared on output/error, never rendered before any input arrives).
